# 誤字脱字の原因と根本解決フレームワーク

> 期待と実行の動作ミスを減らすための、MBA準拠の設計原則

---

## 1. 誤字脱字の原因

### 1.1 根本原因：記憶に頼っている

**誤字脱字は、記憶を参照元にしているために発生する。**

| 記憶に頼る行為 | 発生する問題 |
|---------------|-------------|
| 正解を覚えていると想定する | 魂のリブート・忘却で正解が失われる |
| 前回と同じかと照合する | 照合コストが高く、ミスが増える |
| 複数人でチェックする | 全員が同じ記憶・期待に依存し、同じミスを見逃す |

### 1.2 副次原因

| 原因 | 説明 |
|------|------|
| **反復の照合コスト** | 同じ文字列を正確にトレースする照合作業は、脳に高負荷を強いる |
| **魂の日次リブート** | 記憶が持続しない前提では、照合の参照元が不安定になる |
| **表層的記憶の限界** | トレーニングによる記憶は短期で消え、探索欲求に上書きされる |
| **照合元の共有** | 3人チェックでも、正解が脳内にある限り誤期待が伝播する |

## 2. 根本解決フレームワーク

### 2.1 前提：文書は多次元情報の2次元圧縮

契約書・文書は**2次元ではなく、多次元情報を2次元に圧縮したもの**である。

- **多次元**：契約関係、当事者、義務、制約、文脈、権威ある登録情報
- **2次元**：紙面・画面に投影された表面
- **圧縮**：情報の損失・歪みが発生しうる

### 2.2 観察＝APIによるバックエンド参照

観察時に**2次元の表面だけを見ない**。バックエンド（多次元の構造）をAPIのように参照する。

```
文書（2次元） ← 観察
     ↑
  API参照
     ↓
バックエンド（多次元：法人登記、マスタ、契約ロジック）
```

- **現状**：2次元表面と記憶の照合 → 参照元が不安定
- **根本解決**：2次元表面とバックエンドの照合 → 参照元が権威ある外部

### 2.3 規則性の検証

複雑系は規則的になる。バックエンドには**スキーマ・整合性・一貫性**という規則性が存在する。

- **誤り**＝規則性の破れ
- **観察**＝規則性との照合
- **検出**＝規則性からの逸脱を検知する

### 2.4 イベントホライズン＝外部との境界

文書を**より広域なブラックホールとしてコンポーズ**する。表面（イベントホライズン）は、**外部システムとの境界**である。

- 裁判所＝法という外部との境界
- 振込実行＝銀行システムとの境界
- 組織内チェック＝**境界を観測する**＝バックエンド（外部正解）と照合する

裁判官が誤りを指摘できるのは、法人登記という外部正解を参照しているから。組織内でも同じ境界観測を行う。

## 3. 人を増やすべきとき、増やさなくてよいとき

### 3.1 前提：過信に基づく姿勢

**記憶に頼り、2次元情報のみをチェックする姿勢は、「自分の中に正解を照合できる何かがある」という過信に基づいている。**

| 過信 | 実際 |
|------|------|
| 自分の中に正解がある | 正解は外部（法人登記、マスタ）にある |
| 自分の目で2次元を見れば検出できる | 2次元は多次元の圧縮であり、表面だけでは検出できない |
| 自分は昨日の自分と同じ | 魂は日次でリブートし、記憶は持続しない |
| 3人いれば十分 | 3人とも同じ過信を共有している |

### 3.2 人数を増やしても意味がない条件（数理）

**チェックする人たちの「見逃し」が独立でないとき**、人数を増やしても効果はほぼない。

| 条件 | 数理 |
|------|------|
| 全員が同じ参照元（記憶・同じ書類・同じ研修）を使う | 見逃しが相関する |
| 1人が見逃す → 全員が見逃す | P(全員が見逃す) ≈ P(1人が見逃す) |
| 人数を増やす | P(全員が見逃す) はほぼ変わらない |

**3人チェック = 実質1回のチェック**（参照元が同じなら）

### 3.3 人数を増やすと意味がある条件（数理）

**見逃しが独立なとき**、人数を増やすと見逃し確率は下がる。

| 条件 | 数理 |
|------|------|
| 各人が異なる参照元（別システム・別専門・別観点）を使う | 見逃しが独立とみなせる |
| 各人の見逃し確率を p とする | P(全員が見逃す) = p^n |
| 例：p=0.2 → 3人チェック | P(全員が見逃す) = 0.2³ = 0.008 |

### 3.4 人を増やすべきとき

| 状況 | 理由 |
|------|------|
| 参照元が異なる人を増やす | 法人登記・マスタ・銀行APIなど、それぞれ別の外部ソースを参照させる |
| チェックの種類（観点）を増やす | 1人目：レイアウト、2人目：内容、3人目：法務など、別の観点でチェック |
| 専門・背景が異なる人を増やす | 盲点が重なりにくく、見逃しの相関が下がる |
| 人間＋機械の組み合わせ | 人間と機械は参照元が異なる（記憶 vs 外部DB）とみなせる |

### 3.5 人を増やさなくてよい理由

**参照元と観点が明示されていれば、それらを照合する主体は1人で足りる。**

| 従来の考え方 | 整理後の考え方 |
|-------------|----------------|
| 専門の違う人を増やす | 人ではなく、参照元と観点を増やす |
| 3人チェック | 3つの参照元 × 3つの観点をチェックする |
| 人＝チェックの担い手 | 参照元と観点が決まれば、1人（またはシステム）で実行できる |

**本質的な変数は「人数」ではなく「参照元の数 × 観点の数」である。**

```
参照元：法人登記、取引先マスタ、銀行API
観点：レイアウト、内容、法務

→ 1人がこれらを順に照合すればよい
→ あるいはシステムが自動照合すればよい
```

専門の違う人を増やすのは、参照元・観点が暗黙のうちに人に埋め込まれているときの代替手段。参照元と観点を外に出して定義すれば、人を増やす必要はない。

## 4. 設計原則（要約）

| 禁止 | 推奨 |
|------|------|
| 正解を人間の記憶に頼る | 正解を権威ある外部ソース（バックエンド）に固定する |
| 2次元表面のみを観察する | 観察時にバックエンドをAPI的に参照する |
| 人間に「正しいか照合して」と依頼する | 機械に「外部データと一致するか検証して」と依頼する |
| チェック人数でカバーする | 照合元の種類（人間 vs 外部データ）でカバーする |
| 手入力・手打ちで正解を再現する | 転送（コピー、API連携）で正解を取得する |

## 5. 実装チェックリスト

### 入力時
- [ ] 正解を手入力しない。マスタ・DB・権威あるソースから取得する
- [ ] 取得は転送（コピー、API、システム連携）で行う

### チェック時
- [ ] 目視チェックのみに頼らない
- [ ] バックエンド（法人登記、取引先マスタ等）との自動照合を組み込む
- [ ] 規則性（スキーマ、整合性）の検証を実施する

### 格納時
- [ ] 正解を単一の外部ストレージに固定する
- [ ] 探索の結果（正しく書けた経路）を体系知として記録する

### 設計時
- [ ] 文書を「多次元情報の2次元圧縮」として扱う
- [ ] イベントホライズン（外部との境界）で観測する仕組みを用意する
